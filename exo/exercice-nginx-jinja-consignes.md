# Exercise — Ansible variables and Jinja2 template (`nginx.conf`)

## Objective

Deploy **Nginx** on the three **web** nodes of the Docker lab by driving configuration through **group variables** and an **`nginx.conf` file generated by a Jinja2 template** (`.j2`).

**Estimated duration**: 60 to 90 minutes.

---

## Prerequisites

- Repository Docker lab started:

```bash
docker compose -f docker-compose-lab.yml up -d
```

- Wait for containers to finish initial setup (SSH, base packages), then verify hosts `ansible-lab-web01`, `ansible-lab-web02`, `ansible-lab-web03`.
- Ansible installed on the machine running playbooks (`ansible-playbook --version`).

---

## Useful reminders (without giving away the solution)

- An Ansible **template** is a file on the controller (often in `templates/`) rendered with **Jinja2**, then copied to the target (find in Ansible docs the module that deploys a file from a template).
- Variables for an inventory **group** usually live in **`group_vars/`**, in a file named after the group (by convention).
- In a template, use Jinja2 **conditions** and **loops** to avoid duplicating configuration blocks.
- The **`ansible_managed`** directive (header comment in generated files) marks a file as Ansible-managed; see how to insert it cleanly in a template.

---

## Mission 1 — Inventory

Create an inventory (YAML recommended) that:

- defines a group for the three lab web servers (`web01`, `web02`, `web03`);
- for each host, sets the **Docker container name** expected by Ansible with `docker` connection (as in the repo intro exercises);
- applies connection variables required by the lab (`ansible_connection`, user, Python interpreter).

You may work in a dedicated folder (for example `my-nginx-jinja-exercise/`).

---

## Mission 2 — Group variables

For the web server group, define at least variables for:

- Nginx HTTP **listen port** (make it configurable rather than hard-coded in the template);
- **`server_name`** (ideally **different per machine**: a variable can use facts or inventory hostname);
- document **root directory** for served files;
- a **list** of index file names (used in `nginx.conf` via a Jinja2 loop);
- a boolean such as “enable gzip or not” and, if gzip is enabled, a **list of MIME types** for `gzip_types`;
- a **max** client body size (`client_max_body_size` or equivalent);
- a **dictionary** of HTTP headers to add (name → value), deployed via a **loop** over the dictionary in the template.

Choose readable variable names consistent with the template.

---

## Mission 3 — `nginx.conf.j2` template

Produce a **valid** `nginx.conf` for Ubuntu (official `nginx` package), using a template that:

- includes `mime.types` from the package (usual path under `/etc/nginx/`);
- uses your variables for `worker_connections`, listen port, `server_name`, `root`, `index`, conditional gzip, `client_max_body_size`, and extra headers;
- stays **minimal but functional**: a single `server` block is enough for this exercise.

**Teaching constraint**: at least one **`{% if ... %}` condition** and at least one **`{% for ... %}` loop** must appear in the template (in addition to usual `{{ variable }}` usage).

---

## Mission 4 — Playbook

Write a playbook that, on the web group:

- installs the **nginx** package (container distribution package manager);
- ensures the document root exists and places a **minimal home page** (static or generated HTML);
- deploys your template to **`/etc/nginx/nginx.conf`**, validating syntax before replacing the final file (the template module has a **validate** option; check the docs);
- applies the new configuration: in a container without classic *systemd*, think about how to **reload or restart** Nginx reliably (avoid “magic” solutions you do not understand).

---

## Success criteria (self-assessment)

On **each** container `ansible-lab-web01` through `ansible-lab-web03`:

- `nginx -t` completes without error;
- an HTTP request to **your configured port** on `127.0.0.1` **inside** the container returns **200** (for example with Python `urllib` or a tool you install via the playbook).

---

## Common pitfalls (think, not the solution)

- **Port 80** may already be used if you ran other playbooks on the same lab: a listen-port variable helps.
- Ansible interprets **Jinja2** in YAML playbooks: some characters in `shell` strings can be ambiguous; Ansible docs and the `command` module can simplify tests.
- After replacing `nginx.conf` entirely, **PID file** or leftover processes can break a simple “reload”; anticipate this in a container environment.
